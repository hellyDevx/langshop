# Data Model

## 1. Current Schema

The existing Prisma schema (`prisma/schema.prisma`) contains 5 models powering the prototype. Database is SQLite for development with PostgreSQL as the production target.

### Session

Shopify session management (auto-generated by `@shopify/shopify-app-remix`).

```prisma
model Session {
  id                  String    @id
  shop                String
  state               String
  isOnline            Boolean   @default(false)
  scope               String?
  expires             DateTime?
  accessToken         String
  userId              BigInt?
  firstName           String?
  lastName            String?
  email               String?
  accountOwner        Boolean   @default(false)
  locale              String?
  collaborator        Boolean?  @default(false)
  emailVerified       Boolean?  @default(false)
  refreshToken        String?
  refreshTokenExpires DateTime?
}
```

### TranslationJob

Tracks bulk/auto-translation job execution.

```prisma
model TranslationJob {
  id             String   @id @default(uuid())
  shop           String
  status         String   @default("pending")   // pending | running | completed | failed | partially_failed
  provider       String                          // google | deepl | claude | openai
  resourceType   String
  sourceLocale   String
  targetLocale   String
  marketId       String?
  totalItems     Int      @default(0)
  completedItems Int      @default(0)
  failedItems    Int      @default(0)
  errorMessage   String?
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt

  entries TranslationJobEntry[]

  @@index([shop, status])
  @@index([shop, createdAt])
}
```

### TranslationJobEntry

Per-field translation results within a job.

```prisma
model TranslationJobEntry {
  id              String   @id @default(uuid())
  jobId           String
  resourceId      String
  key             String
  sourceValue     String
  translatedValue String?
  status          String   @default("pending")  // pending | completed | failed
  errorMessage    String?
  createdAt       DateTime @default(now())

  job TranslationJob @relation(fields: [jobId], references: [id], onDelete: Cascade)

  @@index([jobId, status])
}
```

### Known Issues

1. **Plaintext API keys** — `TranslationProviderConfig.apiKey` stored as plaintext. Must add encryption at rest before any public app distribution.
2. **Missing `write_metafields` scope** — Image metafield operations require this scope but it's not declared in `shopify.app.toml`.
3. **API version** — Codebase may reference older API versions; target is `2026-04`.
4. **AppDistribution** — `AppDistribution.AppStore` in config contradicts private app intent.

### TranslationSuggestion

AI-generated translation suggestions awaiting merchant review.

```prisma
model TranslationSuggestion {
  id              String    @id @default(uuid())
  shop            String
  resourceId      String
  resourceType    String
  fieldKey        String
  locale          String
  marketId        String?                    // Nullable for global translations
  sourceValue     String                     // Original content at time of suggestion
  suggestedValue  String                     // AI-generated suggestion
  editedValue     String?                    // Merchant's edited version (null if accepted as-is)

```

## 5. Migration Strategy

### Phase 1: Independent New Models (No FK Conflicts)

Add models that have no foreign key dependencies on existing models:
- `GlossaryTerm`
- `BrandVoiceConfig`
- `OnboardingState`

These can be created in a single migration with zero risk to existing data.

### Phase 2: Logging & Alert Models

Add models that reference existing job IDs (soft reference, not FK):
- `TranslationAuditLog`
- `TranslationAlert`

No foreign key constraints to existing models — `jobId` is stored as a string reference, not a Prisma relation. This allows alerts to persist even if the originating job is deleted.

### Phase 3: Analytics & Sync Models

Add models for analytics and content synchronization:
- `UsageTracking`
- `ContentDigest`
- `TranslationSuggestion`

These are fully independent models populated by new service code.

### Phase 4: Modify Existing Models

Add new fields with defaults to existing models (non-breaking):
- `TranslationJob` — add `retryCount`, `scheduledAt`, `startedAt`, `completedAt`, `glossaryApplied`
- `TranslationJobEntry` — add `providerResponse`
- `TranslationProviderConfig` — add `displayName`, `monthlyQuota`, `quotaUsed`, `quotaResetDate`
- `TranslationStats` — add `marketId`, update unique constraint

All new fields have defaults or are nullable, so existing rows are unaffected.

### Production Migration: SQLite → PostgreSQL

When moving to production/public launch:
1. Export SQLite data via Prisma's migration tooling
2. Update `datasource db` in `schema.prisma` to PostgreSQL connection string
3. Run `prisma migrate deploy` against PostgreSQL
4. Import data with type conversions (SQLite integers → PostgreSQL bigint where needed)
5. Validate with `prisma db pull` to confirm schema matches

**Key differences to handle:** SQLite doesn't enforce string lengths; PostgreSQL does. Review all `String` fields for appropriate `@db.VarChar(n)` or `@db.Text` annotations before production.

## 6. Database Considerations

### Indexing Strategy

Every model that stores per-shop data has a composite index on `shop` + the most frequently queried field. This ensures multi-tenant queries are efficient:

| Model | Index | Purpose |
|-------|-------|---------|
| TranslationJob | `[shop, status]` | Active jobs per shop |
| TranslationJob | `[shop, createdAt]` | Job history |
| TranslationJobEntry | `[jobId, status]` | Entry status per job |
| TranslationAlert | `[shop, dismissed, createdAt]` | Active alerts per shop |
| TranslationAuditLog | `[shop, resourceId, locale]` | Audit trail per resource |
| TranslationAuditLog | `[shop, createdAt]` | Audit timeline |
| UsageTracking | `[shop, date]` | Usage per date range |
| ContentDigest | `[shop, resourceId]` | Digest lookup per resource |
| GlossaryTerm | `[shop, sourceLocale]` | Terms per locale |
| TranslationSuggestion | `[shop, status]` | Pending suggestions |
| TranslationSuggestion | `[shop, resourceId, locale]` | Suggestions per resource |

### Data Retention

| Model | Retention | Rationale |
|-------|----------|-----------|
| TranslationAuditLog | 90 days | Balance between audit trail and storage |
| TranslationAlert | 30 days (dismissed) | Keep active alerts indefinitely |
| UsageTracking | 365 days | Annual reporting |
| TranslationSuggestion | 90 days (rejected) | Keep accepted suggestions indefinitely |
| ContentDigest | Indefinite | Needed for ongoing change detection |

Implement retention via a scheduled cleanup service (daily cron or background job).

### Soft Delete

Not implemented for V1. Hard delete with audit log is sufficient:
- Translations are registered with Shopify (source of truth is Shopify, not our DB)
- Job records serve as operational history, not source of truth
- Audit log preserves the "what happened" regardless of record deletion

### Encryption

API keys in `TranslationProviderConfig.apiKey` require encryption before public launch:
- Use AES-256-GCM with a key stored in environment variables
- Encrypt on write, decrypt on read in the service layer
- V1 (private app): plaintext is acceptable for development velocity
- Pre-launch: add `encryptedApiKey` field, migrate data, remove `apiKey`
